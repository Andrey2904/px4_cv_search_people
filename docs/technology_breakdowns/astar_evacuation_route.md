# A* планирование эвакуационного и возвратного маршрута

## Назначение

В проекте A* используется не для первичного поиска человека, а для
возвратного маршрута после завершения визуального подлета к найденной цели.
Основной сценарий находится в `src/offboard_takeoff/offboard_takeoff/node.py`:

1. квадрокоптер летит по поисковому маршруту;
2. YOLO-детектор подтверждает человека;
3. миссионный узел переходит в `FOLLOW_PERSON`;
4. после визуального подлета вызывается `_finish_person_action()`;
5. узел готовит возврат домой через `_prepare_return_home_waypoint()`;
6. затем, если разрешен A*, вызывает `_prepare_evacuation_route()`;
7. состояние переводится в `RETURN_HOME`.

По фактическому коду маршрут называется `evacuation_route`, но в текущей
реализации он ведет от текущей позиции БПЛА к домашней точке (`home_position`),
а не к отдельной физической точке эвакуации пострадавшего. После достижения
дома миссия может выполнить телепорт настроенной модели человека вниз в Gazebo
и продолжить поиск. Поэтому в дипломе корректнее описывать этот блок как
"планирование безопасного возвратного маршрута после фиксации цели" или как
"эвакуационный/возвратный маршрут в рамках имитационной логики спасения".

## Место A* в автомате миссии

Возвратный маршрут создается в `_finish_person_action()` после завершения
действий с найденным человеком:

```python
self.pending_search_resume_after_home = True
self.pending_teleport_after_home = True
self._prepare_return_home_waypoint()
self._prepare_evacuation_route()
self.set_state(MissionState.RETURN_HOME, ...)
```

Сначала `_prepare_return_home_waypoint()` создает прямую целевую точку возврата:

```python
x = home_position.x
y = home_position.y
z = mission_plan.takeoff_waypoint.z
yaw = текущий yaw цели или yaw домашней точки
```

Это важно: прямой `return_home_waypoint` создается всегда, если есть
`home_position`. A* является улучшением поверх этого прямого возврата. Если
маршрут A* не получится построить, миссия не останавливается, а продолжает
использовать прямой return-home.

После построения A* маршрут хранится в:

```python
self.evacuation_route_waypoints
self.evacuation_route_index
self.return_home_waypoint
```

Первый waypoint A* становится активным `return_home_waypoint`. В состоянии
`RETURN_HOME` метод `handle_return_home()` обновляет сглаженную цель и проверяет
достижение текущей точки. Если точка достигнута, `_advance_evacuation_route()`
выбирает следующую. Когда точки закончились, миссия либо возвращается к поиску,
либо переходит к посадке.

## PlannerConfig

`PlannerConfig` - dataclass с параметрами планировщика. Он используется и для
возвратного A*, и как базовый класс для автоматического покрытия зоны поиска.
Для эвакуационного маршрута в `node.py` заполняются только параметры,
относящиеся к `EvacuationRoutePlanner`.

Поля конфигурации:

| Поле | Значение по умолчанию в `PlannerConfig` | Значение по умолчанию в `node.py` для эвакуации | Назначение |
|---|---:|---:|---|
| `world_sdf_path` | обязательное поле | `/home/dron/PX4-Autopilot/Tools/simulation/gz/worlds/forest_big.sdf` | Путь к SDF-файлу мира Gazebo. |
| `grid_resolution` | `1.0` | `1.0` | Размер клетки occupancy grid в метрах. |
| `obstacle_margin` | `3.0` | `3.0` | Дополнительный отступ вокруг препятствий. |
| `map_padding` | `8.0` | `8.0` | Запас вокруг стартовой точки, цели и препятствий при построении сетки. |
| `waypoint_spacing` | `3.0` | `3.0` | Минимальное расстояние между итоговыми waypoint-ами после прореживания. |
| `coverage_row_spacing` | `7.0` | для эвакуации не задается отдельно | Используется планировщиком поиска, не A* возврата. |
| `mission_rotation_deg` | `90.0` | `90.0` | Поворот между системой координат карты/мира и mission-local XY. |
| `mission_invert_y` | `True` | `True` | Инверсия Y при переходе в mission-local XY. |
| `max_iterations` | `20000` | `20000` | Верхний лимит итераций A*. |

В `node.py` этим параметрам соответствуют ROS 2 параметры:

```text
enable_evacuation_astar
evacuation_world_sdf_path
evacuation_grid_resolution
evacuation_obstacle_margin
evacuation_map_padding
evacuation_waypoint_spacing
evacuation_mission_rotation_deg

evacuation_max_iterations
```

Если `enable_evacuation_astar=False`, `_prepare_evacuation_route()` сразу
возвращается и миссия использует прямой `return_home_waypoint`.

## Чтение SDF и выделение препятствий

Карта препятствий строится при создании `EvacuationRoutePlanner`:

```python
self.obstacles = self._load_obstacles()
```

Метод `_load_obstacles()` читает файл из `PlannerConfig.world_sdf_path`.
Файл раскрывается через `Path(...).expanduser()`. Если файла нет, планировщик
возвращает пустой список препятствий. Если XML не удалось разобрать, также
возвращается пустой список.

Перед XML-парсингом используется `_read_world_sdf_xml()`. В ней есть небольшая
практическая правка: если в SDF встречается malformed-комментарий вида
`<!--- ... --->`, он заменяется на стандартный XML-комментарий `<!-- ... -->`.
Это сделано для устойчивости к конкретному формату мира.

Дальше код ищет элемент:

```xml
<world>
```

и проходит по его дочерним:

```xml
<include>
```

Для каждого include извлекаются:

```text
uri
name
pose
```

Имя модели берется из `model://...` через `_model_name_from_uri()`. Например,
из `model://house` получится `house`.

## Какие модели считаются препятствиями

Планировщик не загружает геометрию моделей из файлов `.sdf` или mesh. Вместо
этого в `path_planner.py` задан словарь известных footprint-ов:

```python
MODEL_FOOTPRINTS = {
    'house': (16.27, 6.07),
    'destroy_building': (30.58, 28.97),
    'roof_rubble': (3.015, 3.015),
    'rubble': (10.79, 4.09),
    'mango_tree': (4.225, 4.225),
}
```

Только модели из этого словаря становятся препятствиями. Если модель есть в
SDF, но ее имени нет в `MODEL_FOOTPRINTS`, A* ее игнорирует.

Отдельно задан список исключаемых имен:

```python
EXCLUDED_MODEL_NAMES = {
    'grave',
    'man',
    'men',
    'person',
    'people',
    'victim',
    'walker',
}
```

Метод `_is_excluded()` проверяет имя модели и имя экземпляра, разбивая строку
по неалфавитным символам. Это нужно, чтобы люди, пострадавшие, прохожие и
некоторые декоративные объекты не воспринимались как препятствия для маршрута.
Также `grave` исключается явно.

## Инфляция препятствий

Каждое препятствие представляется не точкой, а прямоугольным полигоном в
плоскости XY. Размер берется из `MODEL_FOOTPRINTS`, а затем увеличивается на
`obstacle_margin` с каждой стороны:

```python
half_x = size_x / 2.0 + obstacle_margin
half_y = size_y / 2.0 + obstacle_margin
```

После этого строятся четыре локальных угла:

```text
(-half_x, -half_y)
( half_x, -half_y)
( half_x,  half_y)
(-half_x,  half_y)
```

Затем они поворачиваются на yaw из SDF pose и переносятся в позицию модели.
Фактически используется ориентированный прямоугольник, расширенный на запас
безопасности. Такой запас нужен, потому что квадрокоптер имеет ненулевые
габариты, GPS/локальная позиция и управление не идеальны, а полет по waypoint-ам
не гарантирует прохождение строго через математическую линию.

Важно: инфляция выполняется на уровне footprint-а, а не морфологической
операцией по уже построенной occupancy grid. То есть сначала расширяется
геометрический прямоугольник препятствия, и только потом точки сетки проверяются
на попадание внутрь него.

## Преобразование Gazebo/world -> mission XY

SDF хранит позы моделей в координатах мира Gazebo. Миссионный узел управляет
PX4 в локальной системе координат, где `VehicleLocalPosition.x/y` и waypoint-ы
используются как mission-local XY. Чтобы карта препятствий совпала с
координатами миссии, углы препятствий переводятся методом
`_map_to_mission_xy()`.

В `path_planner.py` этот метод описан как обратное преобразование к
`_mission_to_map_xy()` из `top_down_map_viewer.py`. Фактическая формула:

1. берутся координаты SDF/world как `map_x`, `map_y`;
2. выполняется поворот на `-mission_rotation_deg`;
3. если `mission_invert_y=True`, инвертируется результат по Y;
4. возвращается пара `(mission_x, mission_y)`.

Для значений по умолчанию:

```text
mission_rotation_deg = 90.0
mission_invert_y = True
```

это согласуется с настройками viewer-а:

```text
mission_rotation_deg = 90.0
mission_invert_y = True
```

В `top_down_map_viewer.py` визуализация делает обратное: полученные через
`mission/evacuation_route` точки mission-local XY декодируются и переводятся в
карту через `_mission_to_map_xy()`. Поэтому один и тот же маршрут может быть
использован и для управления, и для отрисовки поверх объектов из SDF.

Ограничение: в `EvacuationRoutePlanner` учитываются только `mission_rotation_deg`
и `mission_invert_y`. Параметры viewer-а `mission_swap_xy`, `mission_invert_x`,
`mission_offset_x`, `mission_offset_y` в A* не используются. По текущим
значениям по умолчанию это не конфликтует, потому что swap, invert_x и offsets
по умолчанию выключены/нулевые.

## Построение occupancy grid

Метод `plan(start, goal)` получает старт и цель уже в mission-local XY:

```python
start = (vehicle_local_position.x, vehicle_local_position.y)
goal = (return_home_waypoint.x, return_home_waypoint.y)
```

Дальше вычисляются границы сетки:

```python
bounds = self._compute_bounds(start, goal)
```

`_compute_bounds()` собирает все X/Y координаты:

- старт;
- цель;
- все углы всех препятствий.

После этого добавляет padding:

```python
padding = max(map_padding, obstacle_margin)
```

и возвращает:

```text
min_x, max_x, min_y, max_y
```

Размер сетки вычисляется через `grid_resolution`:

```python
cols = ceil((max_x - min_x) / resolution) + 1
rows = ceil((max_y - min_y) / resolution) + 1
```

При этом resolution дополнительно ограничивается снизу:

```python
resolution = max(grid_resolution, 0.2)
```

Каждая клетка сетки хранит `True` или `False`:

```python
occupied[row][col]
```

Для каждой клетки вычисляется ее центр в mission XY:

```python
x = min_x + col * resolution
y = min_y + row * resolution
```

и проверяется `_is_occupied(x, y)`. Точка считается занятой, если она находится
внутри любого inflated-полигона препятствия. Проверка попадания в полигон
реализована методом `_point_in_polygon()` через ray casting.

## Ближайшая свободная клетка

Стартовая и конечная точки могут оказаться внутри inflated-препятствия или
попасть в занятую клетку из-за дискретизации. Поэтому после преобразования точки
в клетку выполняется поиск ближайшей свободной клетки:

```python
start_cell = self._nearest_free_cell(...)
goal_cell = self._nearest_free_cell(...)
```

`_point_to_cell()` использует округление:

```python
col = round((x - min_x) / resolution)
row = round((y - min_y) / resolution)
```

Если исходная клетка в пределах сетки и свободна, она используется сразу.
Иначе `_nearest_free_cell()` расширяет квадратный радиус поиска:

```text
radius = 1, 2, 3, ...
```

и проверяет только клетки на границе текущего квадрата. Первая найденная
свободная клетка возвращается. Если свободной клетки нет, метод возвращает
`None`, а `plan()` возвращает пустой маршрут.

Это поведение важно для практики: маршрут не падает сразу, если текущая позиция
или home из-за margin оказались формально в занятой области. Но это не точный
поиск геометрически ближайшей клетки по евклидову расстоянию: порядок обхода
зависит от вложенных циклов по `d_row` и `d_col`.

## 8-связный A*

Основной поиск выполняется в `_astar(start_cell, goal_cell, occupied)`.

Используется 8-связная сетка. Возможные переходы:

```text
вверх, вниз, влево, вправо: стоимость 1.0
4 диагонали: стоимость sqrt(2)
```

Открытое множество хранится в heap:

```python
open_heap = [(0.0, start)]
```

Для восстановления пути используется:

```python
came_from
```

Для стоимости пути от старта:

```python
cost_so_far
```

На каждой итерации извлекается клетка с минимальным priority. Если она равна
цели, путь восстанавливается через `_reconstruct_path()`. Для каждого соседа
проверяется:

- что клетка внутри сетки;
- что она не занята;
- что новый путь дешевле ранее найденного.

Если путь улучшен, вычисляется:

```python
priority = new_cost + self._cell_distance(next_cell, goal)
```

и сосед добавляется в heap.

Поиск ограничен:

```python
iterations < max_iterations
```

Если heap опустел или превышен лимит итераций, `_astar()` возвращает пустой
список.

## Эвристика

Эвристика реализована методом `_cell_distance()`:

```python
return math.hypot(first[0] - second[0], first[1] - second[1])
```

Это евклидово расстояние между клетками в индексном пространстве `(row, col)`.
Так как стоимость ортогонального шага равна `1.0`, а диагонального -
`sqrt(2)`, такая эвристика естественно согласуется с 8-связной сеткой. Она
оценивает оставшееся расстояние по прямой и направляет A* к цели без перебора
всей свободной области.

В текущем коде эвристика не умножается на `grid_resolution`. Это не меняет
выбор пути внутри сетки, потому что стоимости шагов также заданы в клетках, а
не в метрах.

## Преобразование клеточного пути в waypoint-ы

Если A* нашел путь, список клеток преобразуется обратно в точки mission XY:

```python
raw_points = [
    self._cell_to_point(cell, min_x, min_y, resolution)
    for cell in cell_path
]
```

`_cell_to_point()` выполняет обратное преобразование:

```python
x = min_x + col * resolution
y = min_y + row * resolution
```

На этом этапе путь еще может содержать много соседних точек, буквально по одной
на каждую клетку сетки. Для управления PX4 такой путь слишком подробный, поэтому
после A* применяются два шага постобработки:

1. smoothing через line-of-sight;
2. thinning waypoint-ов по минимальному расстоянию.

## Smoothing line-of-sight

Метод `_smooth_path()` удаляет лишние промежуточные точки, если между более
дальними точками есть прямая видимость без пересечения занятых клеток.

Алгоритм:

1. первая точка становится anchor;
2. планировщик пробует соединить anchor сразу с последней точкой;
3. если отрезок свободен, последняя точка добавляется в smoothed route;
4. если отрезок занят, проверяется предпоследняя, затем предыдущие;
5. найденная самая дальняя достижимая точка становится следующим anchor;
6. цикл продолжается до конца маршрута.

Свободность отрезка проверяет `_line_is_free()`. Он:

- считает длину отрезка;
- выбирает число sample-точек;
- идет по отрезку с шагом примерно `resolution * 0.5`;
- каждую sample-точку переводит в клетку;
- возвращает `False`, если точка вышла за границы сетки или попала в занятую
  клетку.

Это не непрерывная геометрическая проверка полигона, а дискретная проверка по
occupancy grid. Поэтому качество smoothing зависит от `grid_resolution` и
`obstacle_margin`.

## Thinning waypoint-ов

После smoothing вызывается `_thin_waypoints()`. Его задача - оставить точки,
достаточно разнесенные для waypoint-following.

Если точек две или меньше, они возвращаются без изменений. Иначе выбирается
минимальный spacing:

```python
spacing = max(waypoint_spacing, grid_resolution)
```

Первая точка сохраняется всегда. Каждая промежуточная точка добавляется только
если расстояние от последней сохраненной точки не меньше `spacing`. Последняя
точка также сохраняется всегда.

Итоговый маршрут остается ломаной линией в mission-local XY. Высота и yaw на
этом этапе не задаются; они добавляются уже в `node.py`.

## Использование маршрута в node.py

`_prepare_evacuation_route()` строит A* так:

```python
planner = EvacuationRoutePlanner(config)
route_xy = planner.plan(start=start, goal=goal)
```

Где:

```python
start = (
    vehicle_local_position.x,
    vehicle_local_position.y,
)
goal = (
    return_home_waypoint.x,
    return_home_waypoint.y,
)
```

Если найдено меньше двух точек:

```python
if len(route_xy) < 2:
    warning('A* evacuation route was not found, using direct return-home')
    return
```

То есть fallback на прямой return-home встроен в архитектуру: перед A* уже
создан `return_home_waypoint`, а неудачный A* просто не заменяет его.

Если маршрут найден, `node.py` создает список `Waypoint`:

```python
self.evacuation_route_waypoints = [
    Waypoint(x=x, y=y, z=altitude, yaw=return_yaw)
    for x, y in route_xy[1:]
]
```

Первая точка `route_xy[0]` пропускается, потому что она соответствует текущей
позиции БПЛА. Высота для всех точек берется из:

```python
mission_plan.takeoff_waypoint.z
```

Yaw для всех точек берется из заранее подготовленного
`return_home_waypoint.yaw`. То есть A* планирует только 2D XY-путь. Вертикальное
движение и ориентация не являются частью алгоритма A*.

## Публикация mission/evacuation_route

После успешного построения вызывается:

```python
self._publish_evacuation_route(route_xy)
```

Topic:

```text
mission/evacuation_route
```

Тип сообщения:

```text
std_msgs/Float32MultiArray
```

Формат данных:

```text
[x0, y0, x1, y1, x2, y2, ...]
```

Все координаты публикуются в mission-local XY. Важно: публикация происходит
только при успешном A*. Если A* не нашел маршрут и используется прямой
return-home, `mission/evacuation_route` в этом месте не публикуется.

В `top_down_map_viewer.py` подписка создается так:

```python
self.create_subscription(
    Float32MultiArray,
    'mission/evacuation_route',
    self._evacuation_route_callback,
    10,
)
```

Callback декодирует пары `(x, y)` и переводит их в map frame через
`_mission_to_map_xy()`. Затем `_draw_evacuation_route()` рисует маршрут желтой
линией с круглыми маркерами. Отрисовку можно выключить параметром:

```text
draw_evacuation_route
```

## Fallback на прямой return-home

В коде есть несколько случаев, когда A* не используется:

- `enable_evacuation_astar=False`;
- нет `vehicle_local_position`;
- нет `home_position`;
- не подготовлен `return_home_waypoint`;
- SDF-файл не найден или не распарсен;
- стартовая или конечная свободная клетка не найдена;
- A* не нашел путь;
- превышен `max_iterations`;
- после smoothing/thinning результат содержит меньше двух точек.

При этом `_prepare_return_home_waypoint()` вызывается до A*, поэтому fallback
сводится к прямому полету в home XY на рабочей высоте. В логе при неуспехе
планировщика появляется предупреждение:

```text
A* evacuation route was not found, using direct return-home
```

Это честная и важная особенность реализации: безопасность возврата не полностью
зависит от планировщика. Но прямой fallback не учитывает препятствия, поэтому в
сложной сцене он может пройти через obstacle footprint.

## Ограничения текущей реализации

1. A* планирует только в 2D.

   Высота постоянная и задается в `node.py` как `mission_plan.takeoff_waypoint.z`.
   Препятствия не имеют высотной модели, а маршрут не выбирает облет сверху.

2. Используются вручную заданные footprint-ы.

   `EvacuationRoutePlanner` не читает реальные размеры моделей из model SDF или
   mesh-файлов. Он знает только имена из `MODEL_FOOTPRINTS`. Остальные модели
   игнорируются.

3. Люди и некоторые декоративные модели исключаются из препятствий.

   Это сделано намеренно, чтобы человек не блокировал возвратный маршрут.
   Однако в реальной задаче вокруг пострадавшего могла бы потребоваться зона
   безопасности.

4. Нет динамических препятствий.

   Occupancy grid строится один раз в момент планирования. Если объект в Gazebo
   переместится после построения маршрута, A* этого не увидит.

5. Нет replanning во время возврата.

   После построения `evacuation_route_waypoints` миссия просто проходит точки.
   Если промежуточная точка становится недостижимой, отдельного повторного A*
   в `handle_return_home()` нет.

6. Преобразование координат ограничено частью параметров viewer-а.

   A* учитывает `mission_rotation_deg` и `mission_invert_y`, но не использует
   swap, invert_x и offsets. Для текущих настроек по умолчанию это согласовано,
   но при изменении viewer transform это нужно синхронизировать вручную.

7. Занятость клетки определяется по центру клетки.

   Если край препятствия проходит между центрами клеток, точность зависит от
   `grid_resolution` и `obstacle_margin`.

8. Smoothing дискретный.

   `_line_is_free()` проверяет набор точек на отрезке, а не непрерывное
   пересечение отрезка с полигонами.

9. Fallback прямой и не obstacle-aware.

   Если A* не построился, БПЛА возвращается напрямую к home waypoint.
   Это повышает живучесть миссии, но не гарантирует обход препятствий.

10. `mission/evacuation_route` - диагностический/визуализационный topic.

   Управление БПЛА идет не подпиской на этот topic, а внутренним списком
   `evacuation_route_waypoints` в `node.py`.

## Что можно написать в дипломе

В программной системе реализован модуль построения возвратного маршрута на
основе алгоритма A*. После обнаружения и визуальной фиксации цели БПЛА должен
вернуться в домашнюю точку перед продолжением поиска или завершением миссии.
Для этого текущая позиция аппарата и домашняя точка переводятся в двумерную
mission-local систему координат, а статические препятствия извлекаются из SDF
описания мира Gazebo. Каждое известное препятствие представляется
ориентированным прямоугольным footprint-ом, расширенным на заданный безопасный
отступ. По полученным полигонам строится occupancy grid, где занятые клетки
соответствуют областям препятствий.

Поиск пути выполняется на 8-связной сетке. Стоимость ортогонального перехода
равна единице, стоимость диагонального перехода равна `sqrt(2)`, а в качестве
эвристики используется евклидово расстояние между клетками. После нахождения
клеточного пути выполняется сглаживание с проверкой прямой видимости по
occupancy grid, а затем прореживание waypoint-ов по минимальному расстоянию.
Итоговый 2D-маршрут преобразуется в набор waypoint-ов PX4 с постоянной рабочей
высотой и публикуется в `mission/evacuation_route` для визуализации на
top-down карте.

Если A* не может построить маршрут, система использует заранее подготовленную
прямую точку возврата домой. Такое решение сохраняет выполнение миссии при
ошибке планировщика, но прямой fallback не учитывает препятствия и поэтому
рассматривается как ограничение текущей симуляционной реализации.
