# Захват и наведение на цель по bounding box

## Назначение

Подсистема захвата цели отвечает за реакцию БПЛА на обнаруженного человека:
миссионный узел получает bounding box от детектора, проверяет, что обнаружение
достаточно надежное, останавливает поисковый маршрут, разворачивает БПЛА по
горизонтальной ошибке bbox и выполняет короткий визуальный подлет вперед.

В текущей реализации это не полноценная оценка расстояния до человека, а
монокулярный визуальный сценарий: близость оценивается косвенно по росту
размера bounding box в кадре и по положению нижней границы bbox относительно
нижнего края изображения.

Основные файлы:

- `src/offboard_takeoff/offboard_takeoff/node.py` - прием bbox, подтверждение
  цели, фазы `FOLLOW_PERSON`, команды движения и завершение действия;
- `src/offboard_takeoff/offboard_takeoff/yolo_detector.py` - публикация
  лучшего bbox цели в ROS 2;
- `README.md` и `docs/technology_breakdowns/mission_state_machine.md` -
  высокоуровневое описание миссии.

## Интерфейс `yolo/target_bbox`

Ключевой интерфейс между компьютерным зрением и миссионной логикой - topic:

```text
yolo/target_bbox
```

Тип сообщения:

```text
std_msgs/Int32MultiArray
```

Формат массива:

```text
[confidence_milli, center_x, center_y, width, height, image_width, image_height]
```

Поля:

| Поле | Смысл |
| --- | --- |
| `confidence_milli` | Уверенность детектора, умноженная на `1000`. Например, `870` соответствует `0.87`. |
| `center_x` | Горизонтальная координата центра bbox в пикселях. |
| `center_y` | Вертикальная координата центра bbox в пикселях. |
| `width` | Ширина bbox в пикселях. |
| `height` | Высота bbox в пикселях. |
| `image_width` | Ширина кадра в пикселях. |
| `image_height` | Высота кадра в пикселях. |

В `yolo_detector.py` этот массив строится из лучшей целевой детекции:

```text
confidence * 1000,
(x1 + x2) / 2,
(y1 + y2) / 2,
x2 - x1,
y2 - y1,
frame_width,
frame_height
```

Если целевой детекции нет, YOLO-детектор публикует пустой массив `[]`. В
`node.py` сообщение короче 7 элементов сбрасывает текущую цель:
`person_target = None`.

## Выбор целевой детекции в детекторе

Детектор публикует не все bbox для наведения, а один лучший bbox цели.
В `yolo_detector.py` список детекций сортируется по confidence, а затем
`_best_target_detection()` выбирает детекцию с максимальной уверенностью среди
меток из `target_labels`. По умолчанию целевая метка - `person`.

Параллельно детектор публикует и вспомогательные topic-и:

```text
yolo/target_detected
yolo/detection_count
yolo/class_ids
yolo/bounding_boxes
yolo/class_labels
yolo/debug_image
```

Для наведения в `node.py` используется именно `yolo/target_bbox`.

Важно: `target_detected` имеет собственную логику удержания статуса через
`detection_hold_sec`, но миссионная логика захвата цели опирается на свежий
`target_bbox`, а не на один только boolean-флаг.

## `PersonTarget` в миссионном узле

В `node.py` входной массив преобразуется в dataclass `PersonTarget`:

```text
confidence: float
center_x: int
center_y: int
width: int
height: int
image_width: int
image_height: int
```

Дополнительно класс вычисляет нормированные величины:

```text
center_x_error = (center_x - image_width / 2) / (image_width / 2)
height_ratio = height / image_height
width_ratio = width / image_width
left_ratio = (center_x - width / 2) / image_width
right_ratio = (center_x + width / 2) / image_width
bottom_ratio = (center_y + height / 2) / image_height
```

`center_x_error` - главная ошибка наведения по yaw. Если значение близко к
нулю, центр bbox расположен около центра кадра. Отрицательное или
положительное значение означает смещение цели влево или вправо относительно
оптического центра изображения.

`bottom_ratio` используется как защитный критерий остановки: если нижний край
bbox почти дошел до нижнего края кадра, дрон прекращает движение вперед.

## Фильтрация ошибки и память цели

После приема bbox миссионный узел обновляет сглаженную горизонтальную ошибку:

```text
filtered_error = alpha * raw_error + (1 - alpha) * previous_filtered_error
```

Параметр:

```text
follow_error_filter_alpha = 0.25
```

Эта фильтрация уменьшает резкие изменения команды yaw, возникающие из-за
дрожания bbox от кадра к кадру. При этом в фазе `ALIGN_TO_TARGET` текущая
логика поворота использует `person_target.center_x_error` напрямую, а
сглаженная ошибка также сохраняется для логов и вспомогательных функций.

Если абсолютная ошибка больше deadband, она сохраняется как
`last_significant_person_center_x_error`. Кроме того, если цель была видима
дольше:

```text
follow_target_memory_activation_sec = 1.0
```

то последний стабильный `PersonTarget` записывается в память:

```text
last_stable_person_target
last_stable_person_target_at
```

Эта память используется при кратковременной потере bbox на этапе
выравнивания. Память считается актуальной в течение:

```text
follow_target_memory_timeout_sec = 1.5
```

Если стабильный bbox уже устарел, но последняя значимая ошибка yaw была больше
deadband, узел может использовать эту последнюю ошибку как запасной ориентир.
Если и запомненная ошибка мала, память не используется.

## Проверка качества bbox

Перед тем как bbox считается пригодным для реакции миссии, `node.py`
проверяет:

1. не включен ли временный cooldown после предыдущего спасения;
2. есть ли текущий `person_target`;
3. не ниже ли confidence заданного порога;
4. достаточно ли велик bbox по ширине и высоте;
5. не устарели ли данные от vision.

Основные параметры:

| Параметр | Значение по умолчанию | Смысл |
| --- | ---: | --- |
| `follow_min_confidence` | `0.70` | Минимальная уверенность bbox для миссионной реакции. |
| `search_min_bbox_width_px` | `40` | Минимальная ширина bbox во время поиска. |
| `search_min_bbox_height_px` | `40` | Минимальная высота bbox во время поиска. |
| `follow_min_bbox_width_px` | `35` | Минимальная ширина bbox во время активного следования. |
| `follow_min_bbox_height_px` | `35` | Минимальная высота bbox во время активного следования. |
| `follow_person_timeout_sec` | `1.0` | Максимальный возраст данных vision для использования bbox. |

Порог размера зависит от состояния. В `SEARCH` и `HOLD` применяются более
строгие `search_min_bbox_*`, а в `FOLLOW_PERSON` - `follow_min_bbox_*`.
Такой выбор позволяет не запускать подлет по слишком мелкой дальней детекции,
но не сбрасывать сопровождение сразу из-за небольших колебаний размера bbox
после входа в режим следования.

## Подтверждение обнаружения

Подлет не начинается от единичного срабатывания. Функция
`_maybe_start_person_follow()` работает только в состояниях:

```text
SEARCH
HOLD
```

Если `enable_person_follow = True` и текущий bbox прошел фильтры качества,
узел начинает окно подтверждения. Цель должна оставаться доступной в течение:

```text
follow_detection_confirm_sec = 1.0
```

Если bbox кратковременно пропадает, подтверждение не сразу сбрасывается.
Допустимый разрыв:

```text
follow_detection_gap_tolerance_sec = 0.5
```

Если цель отсутствует дольше этого интервала, `person_detection_started_at`
сбрасывается, и следующее обнаружение должно заново пройти полное окно
подтверждения.

Пока подтверждение идет, дрон удерживает текущую позицию через
`_hold_current_position_target()`. Это важно: система не продолжает активно
лететь по поисковому waypoint-у, пока проверяет, является ли обнаружение
устойчивым.

После подтверждения:

1. запоминается индекс поискового waypoint-а, откуда нужно будет продолжить
   миссию;
2. состояние меняется на `FOLLOW_PERSON`;
3. запоминается начальная высота bbox:

```text
person_reference_bbox_height = max(person_target.height, 1.0)
```

Эта высота далее используется как базовая величина для критерия приближения.

## Состояние `FOLLOW_PERSON`

Внутри `FOLLOW_PERSON` есть собственная последовательность фаз:

```text
STOP_BEFORE_APPROACH
ALIGN_TO_TARGET
APPROACH_FORWARD
HOLD_AFTER_APPROACH
```

Общий защитный таймаут всего действия:

```text
follow_person_action_timeout_sec = 90.0
```

Если состояние `FOLLOW_PERSON` длится дольше этого времени, миссионный узел
вызывает обработку отказа через `handle_failure()`. В зависимости от настроек
миссии это ведет к возврату домой или `FAILSAFE`.

## Фаза `STOP_BEFORE_APPROACH`

Сразу после входа в `FOLLOW_PERSON` дрон не летит вперед. Он зависает на
текущей позиции:

```text
follow_stop_before_approach_sec = 1.0
```

В коде это сделано для стабилизации перед подлетом: комментарий в
`handle_follow_person()` прямо указывает, что перед запоминанием bbox нужен
стабильный hover. На практике базовая высота bbox уже записана при
подтверждении, но эта пауза снижает влияние переходного движения и рывков
изображения перед выравниванием.

После паузы фаза меняется на `ALIGN_TO_TARGET`.

## Фаза `ALIGN_TO_TARGET`

На этом этапе дрон разворачивается по yaw так, чтобы центр bbox оказался
около центра кадра. Если свежий `PersonTarget` доступен, используется ошибка:

```text
yaw_error = person_target.center_x_error
```

Управление yaw:

```text
yaw_rate_cmd = follow_yaw_kp * yaw_error
yaw_rate_cmd = clamp(yaw_rate_cmd, -max_yaw_rate, max_yaw_rate)
next_yaw = current_heading + yaw_rate_cmd * control_period
```

Параметры:

| Параметр | Значение по умолчанию | Смысл |
| --- | ---: | --- |
| `follow_yaw_kp` | `0.8` | Пропорциональный коэффициент поворота по ошибке bbox. |
| `follow_yaw_kd` | `0.0` | Производная составляющая объявлена, но в текущей фазе `ALIGN_TO_TARGET` не используется. |
| `max_yaw_rate` | `0.5` | Ограничение скорости изменения yaw. |
| `follow_yaw_deadband` | `0.05` | Зона, в которой цель считается достаточно центрированной. |
| `follow_align_dropout_continue_sec` | `1.5` | Сколько можно продолжать выравнивание по памяти при потере bbox. |

Если:

```text
abs(center_x_error) <= follow_yaw_deadband
```

цель считается выровненной, и начинается прямой подлет:
`_start_bbox_growth_approach()`.

Если bbox пропал во время выравнивания, узел пытается получить remembered
error через `_get_align_memory_error()`. Возможны два исхода:

- если запомненной ошибки нет, миссия возвращается к поиску через
  `_resume_search_after_alignment_loss()`;
- если ошибка есть, дрон продолжает поворот по этой ошибке, но только пока
  потеря цели не длится дольше `follow_align_dropout_continue_sec`.

Если текущий поисковый waypoint доступен, возврат после потери цели идет в
`SEARCH`. Если waypoint-а нет, узел удерживает позицию и переходит в
`FAILSAFE`.

## Фаза `APPROACH_FORWARD`

После выравнивания дрон двигается вперед по текущему курсу. Команда не
задает расстояние до человека. Вместо этого на каждом цикле обновляется
позиционный setpoint немного впереди текущей цели управления:

```text
step_distance = follow_forward_speed * control_period
x_next = base_x + cos(heading) * step_distance
y_next = base_y + sin(heading) * step_distance
z_next = current_z
yaw_next = heading
```

Основные параметры:

| Параметр | Значение по умолчанию | Смысл |
| --- | ---: | --- |
| `follow_forward_speed` | `0.35` | Скорость визуального подлета вперед, м/с. |
| `follow_target_lead_limit` | `1.5` | Максимальное опережение setpoint-а относительно текущей позиции. |
| `follow_bbox_growth_target` | `4.0` | Во сколько раз bbox должен вырасти относительно базовой высоты. |
| `follow_bottom_edge_margin_ratio` | `0.08` | Запас до нижнего края кадра для остановки. |
| `follow_approach_dropout_stop_sec` | `0.5` | Сколько можно продолжать подлет без свежего bbox. |
| `follow_approach_max_sec` | `12.0` | Максимальная длительность прямого подлета. |

В начале фазы, если `person_reference_bbox_height` почему-то не задана, узел
берет текущую высоту bbox. Если свежего bbox в этот момент нет, дрон просто
удерживает текущую позицию и не начинает движение вперед.

### Критерий остановки по росту bbox

Основной критерий приближения:

```text
current_bbox_height >= person_reference_bbox_height * follow_bbox_growth_target
```

При значении по умолчанию `follow_bbox_growth_target = 4.0` это означает:
подлет завершается, когда высота bbox стала как минимум в 4 раза больше, чем
в момент подтверждения цели.

Такой критерий не измеряет метры до человека. Он использует предположение,
что при движении камеры к объекту его проекция в изображении растет.

### Критерий остановки по нижнему краю кадра

Дополнительный защитный критерий:

```text
bottom_ratio >= 1.0 - follow_bottom_edge_margin_ratio
```

При `follow_bottom_edge_margin_ratio = 0.08` дрон прекращает прямой подлет,
если нижняя граница bbox находится в нижних 8% изображения. Это защищает от
ситуации, когда цель уходит под камеру и дальнейшее движение вперед может
стать небезопасным или бессмысленным для визуального сопровождения.

### Критерий остановки по timeout

Если фаза `APPROACH_FORWARD` длится дольше:

```text
follow_approach_max_sec = 12.0
```

подлет завершается принудительно, даже если bbox не достиг нужного размера.
Это предотвращает бесконечное движение вперед при ошибочной оценке масштаба,
плохой детекции или неподходящей геометрии сцены.

### Критерий остановки по dropout

Если во время прямого подлета свежий bbox пропал, узел не останавливается
мгновенно. Он запускает таймер потери цели:

```text
person_approach_loss_started_at
```

Остановка выполняется, если потеря длится не меньше:

```text
follow_approach_dropout_stop_sec = 0.5
```

Есть дополнительная проверка: если последняя известная цель уже была около
нижнего края кадра, подлет завершается сразу. Это логично, потому что bbox
мог исчезнуть не из-за случайной ошибки детектора, а потому что человек ушел
за нижнюю границу изображения при сближении.

## Фаза `HOLD_AFTER_APPROACH`

После любого штатного завершения прямого подлета вызывается
`_finish_bbox_growth_approach()`. Эта функция:

1. переводит фазу в `HOLD_AFTER_APPROACH`;
2. сбрасывает таймер потери цели при подлете;
3. удерживает текущую позицию через `_hold_person_follow_position()`;
4. записывает причину завершения в лог.

Далее дрон зависает:

```text
follow_hover_after_approach_sec = 5.0
```

После этой паузы вызывается `_finish_person_action()`.

## Что происходит после завершения подлета

Функция `_finish_person_action()` завершает визуальное взаимодействие с
человеком и переводит миссию к возврату домой перед продолжением поиска.

Фактическая последовательность в текущем `node.py`:

1. выставляются флаги:

```text
pending_search_resume_after_home = True
pending_teleport_after_home = True
```

2. подготавливается waypoint возврата домой через
   `_prepare_return_home_waypoint()`;
3. строится эвакуационный маршрут через `_prepare_evacuation_route()`;
4. состояние меняется на:

```text
RETURN_HOME
```

с причиной:

```text
Person fixed, returning home before resuming search
```

Когда дрон достигает home, вызывается `_resume_search_after_home_reached()`.
Там:

1. включается cooldown обнаружений через `_begin_post_rescue_cooldown()`;
2. если `pending_teleport_after_home = True`, вызывается
   `_teleport_rescue_model_down()`;
3. сбрасываются флаги возврата и телепорта;
4. выбирается сохраненный waypoint поиска;
5. миссия возвращается в `SEARCH`.

Параметр cooldown:

```text
post_rescue_detection_cooldown_sec = 5.0
```

Во время cooldown свежие bbox игнорируются, чтобы система не начала повторное
сопровождение той же цели сразу после эвакуации.

Честное замечание по README: в README упоминается публикация события
`mission/person_rescued`, но в текущем `node.py` отдельный publisher этого
topic-а не используется. Фактическое завершение в коде реализовано через
возврат домой, флаги `pending_*`, cooldown и телепорт модели Gazebo.

## Телепорт модели после возврата домой

Телепорт выполняется не в момент остановки перед человеком, а после возврата
home в `_resume_search_after_home_reached()`. Узел пытается переместить
настроенную модель:

```text
rescue_model_name = man_3
rescue_model_teleport_x = 36.57
rescue_model_teleport_y = 28.2
rescue_model_teleport_z = -10.0
teleport_world_name = forest_big
```

Сначала используется Gazebo service:

```text
gz service -s /world/<world>/set_pose
```

Если service-вариант не сработал, код пробует fallback через topic:

```text
gz topic -t /world/<world>/pose/modify
```

Это симуляционное приближение эвакуации: модель человека перемещается ниже
сцены, чтобы не была повторно обнаружена в зоне поиска.

## Параметры захвата цели

| Параметр | Значение по умолчанию | Назначение |
| --- | ---: | --- |
| `enable_person_follow` | `True` | Включает реакцию миссии на обнаруженного человека. |
| `follow_min_confidence` | `0.70` | Минимальная уверенность детектора. |
| `search_min_bbox_width_px` | `40` | Минимальная ширина bbox для старта реакции из поиска. |
| `search_min_bbox_height_px` | `40` | Минимальная высота bbox для старта реакции из поиска. |
| `follow_min_bbox_width_px` | `35` | Минимальная ширина bbox внутри `FOLLOW_PERSON`. |
| `follow_min_bbox_height_px` | `35` | Минимальная высота bbox внутри `FOLLOW_PERSON`. |

## Ограничения монокулярного bbox-подхода

1. Нет прямого измерения расстояния. Система не получает глубину от камеры и
   не вычисляет 3D-позицию человека. Рост bbox используется как косвенный
   признак приближения.

2. Размер bbox зависит не только от расстояния. На него влияют поза человека,
   частичное перекрытие, наклон камеры, точность детектора, освещение и
   положение объекта в кадре.

3. Bbox может дрожать. Для горизонтальной ошибки используется сглаживание и
   deadband, но сама геометрия bbox остается приближенной.

4. Подлет не строит локальную карту препятствий вокруг человека. Команда
   движения в `APPROACH_FORWARD` идет вперед по текущему курсу до выполнения
   одного из критериев остановки.

5. Завершение эвакуации симуляционное. Человек не захватывается физически:
   после возврата домой настроенная модель Gazebo телепортируется вниз под
   сцену.

6. При нескольких людях используется лучший bbox по confidence среди целевых
   меток. Отдельного устойчивого multi-object tracking с ID в текущем
   интерфейсе `target_bbox` нет.

7. Качество поведения зависит от частоты и задержки vision-данных. Если bbox
   старше `follow_person_timeout_sec`, миссионная логика считает его
   непригодным.

## Формулировка для диплома

В разработанной системе захват пострадавшего реализован как визуальная
реакция на bounding box человека, полученный от нейросетевого детектора.
Детектор публикует в ROS 2 topic `yolo/target_bbox` уверенность, центр и
размер рамки, а также размер исходного изображения. Миссионный узел
преобразует эти данные в структуру `PersonTarget`, проверяет confidence,
минимальный размер bbox и свежесть данных, после чего требует устойчивого
обнаружения в течение заданного времени. Такой этап подтверждения снижает
вероятность реакции на одиночные ложные срабатывания.

После подтверждения цель переводит миссию из поиска в состояние
`FOLLOW_PERSON`. Внутри него БПЛА сначала зависает, затем разворачивается по
горизонтальной ошибке между центром bbox и центром кадра. При кратковременной
потере цели используется память последней устойчивой ошибки, что позволяет
продолжить выравнивание в течение ограниченного времени. После центрирования
цели БПЛА выполняет прямой подлет вперед. Поскольку монокулярная камера не
дает прямой оценки расстояния, завершение подлета определяется по косвенным
критериям: росту высоты bbox относительно начального значения, приближению
нижней границы bbox к нижнему краю изображения, таймауту движения или потере
цели. После завершения подлета БПЛА зависает, строит маршрут возврата домой,
а затем продолжает поисковую миссию после симуляционного удаления найденной
модели человека из сцены.
