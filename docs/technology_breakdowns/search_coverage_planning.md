# Планирование маршрута поиска и покрытия зоны

## Назначение

Подсистема планирования поискового маршрута формирует последовательность
waypoint-ов, по которым квадрокоптер проходит зону поиска до обнаружения
пострадавшего. В проекте есть два варианта такого маршрута:

- статический fallback-маршрут "змейкой" из `mission.py`;
- автоматический маршрут покрытия, который строится по SDF-описанию мира
  Gazebo в `SearchCoveragePlanner`.

Обе версии в итоге приводятся к одному формату: набору `Waypoint` в локальной
системе координат PX4 mission-local XY с высотой `z = -takeoff_height`.

Основные файлы:

```text
src/offboard_takeoff/offboard_takeoff/mission.py
src/offboard_takeoff/offboard_takeoff/path_planner.py
src/offboard_takeoff/offboard_takeoff/node.py
src/offboard_takeoff/offboard_takeoff/top_down_map_viewer.py
README.md
docs/technology_breakdowns/mission_state_machine.md
```

## Место в общей миссии

Миссионный узел `OffboardTakeoff` строит план в конструкторе:

```python
self.search_route_xy = []
self.mission_plan = self._build_mission_plan()
```

Метод `_build_mission_plan()` сначала создает статический план:

```python
fallback_plan = default_mission_plan(self.takeoff_height)
```

Затем, если включен параметр:

```text
enable_auto_search_route = True
```

узел пытается построить автоматический маршрут через `SearchCoveragePlanner`.
Если автоматический маршрут не удалось получить, используется статический
fallback из `mission.py`.

После взлета миссия переходит в состояние `SEARCH`, выбирает первый waypoint
и последовательно двигается по маршруту. Если waypoint имеет `hold_time`,
после его достижения происходит переход в состояние `HOLD`, где дрон
удерживает позицию заданное время. Во время `SEARCH` и `HOLD` параллельно
проверяется наличие подтвержденного обнаружения человека.

## Статический fallback-маршрут из `mission.py`

Функция:

```python
default_mission_plan(takeoff_height)
```

создает маршрут вручную заданной "змейкой". Это резервный вариант, который не
зависит от SDF-карты и работает даже тогда, когда автоматическое построение
невозможно.

В коде заданы границы:

```text
search_min_x = -60.0
search_max_x = -20.0
search_max_y = 30.0
search_min_y = -50.0
search_row_spacing = 7.0
row_hold_sec = 1.5
```

Рабочая высота задается в системе PX4 NED как отрицательная координата:

```python
working_altitude = -abs(takeoff_height)
```

Маршрут строится построчно:

1. вычисляются строки `search_rows_y` от `search_max_y` к `search_min_y` с
   шагом `7.0` м;
2. на первой строке дрон начинает в точке `x = search_max_x`;
3. на четных строках он летит к `search_min_x`, на нечетных обратно к
   `search_max_x`;
4. между строками добавляется вертикальный переход по `y`;
5. на концах строк и переходах выставляется `hold_time = 1.5` с;
6. yaw задается по направлению движения: `0`, `pi`, `-pi/2` или `pi/2`.

В комментарии к `mission.py` явно указано, что маршрут подобран под лесную
область: top-down map использует соответствие вида `mission_x ~= Gazebo world
Y`, `mission_y ~= Gazebo world X`, а деревья занимают приблизительную область
Gazebo `X -46..25` и `Y -56..-25`.

Важно: этот маршрут не проверяет препятствия. Он является заранее подобранным
патрульным маршрутом для текущей сцены.

## Автоматический SearchCoveragePlanner

Автоматическое покрытие реализовано классом:

```python
class SearchCoveragePlanner(EvacuationRoutePlanner)
```

Он наследует от `EvacuationRoutePlanner`, поэтому использует тот же механизм
чтения препятствий из SDF, проверки занятости, сеточного A* и прореживания
waypoint-ов.

Основной метод:

```python
plan_coverage(start: tuple[float, float] | None = None)
```

В `node.py` он вызывается со стартовой точкой, равной точке взлета fallback
плана:

```python
route_start_xy = (0.0, 0.0)
route_xy = planner.plan_coverage(start=route_start_xy)
```

Если маршрут содержит меньше двух точек, узел пишет предупреждение:

```text
Auto search route was not generated, using static mission
```

и возвращается к статическому плану.

## Чтение препятствий из SDF

Загрузка препятствий находится в методе:

```python
_load_obstacles()
```

Алгоритм:

1. путь к SDF берется из `PlannerConfig.world_sdf_path`;
2. файл читается как XML;
3. если в файле встречается известная некорректная форма комментариев
   `<!--- ... --->`, она заменяется на корректную `<!-- ... -->`;
4. из корневого XML берется узел `<world>`;
5. перебираются все прямые дочерние `<include>`;
6. из каждого include читаются `uri`, `name` и `pose`;
7. из `model://...` извлекается имя модели;
8. модель отбрасывается, если она относится к людям или явно исключена;
9. для известных моделей берется footprint из словаря `MODEL_FOOTPRINTS`;
10. по pose, yaw и размеру footprint строится повернутый прямоугольник
    препятствия;
11. прямоугольник дополнительно расширяется на `obstacle_margin`;
12. углы переводятся из координат карты Gazebo в mission-local XY.

Если файл не найден, XML не парсится, узел `<world>` отсутствует или ни одна
модель не получила footprint, список препятствий будет пустым. Для
`SearchCoveragePlanner` пустой список препятствий означает невозможность
построить автоматический маршрут покрытия:

```python
if not self.obstacles:
    return []
```

## MODEL_FOOTPRINTS

В планировщике размеры препятствий заданы вручную:

```python
MODEL_FOOTPRINTS = {
    'house': (16.27, 6.07),
    'destroy_building': (30.58, 28.97),
    'roof_rubble': (3.015, 3.015),
    'rubble': (10.79, 4.09),
    'mango_tree': (4.225, 4.225),
}
```

Это важное ограничение: `SearchCoveragePlanner` не вычисляет размеры моделей
из их `model.sdf`, collision geometry или mesh-файлов. Он использует только
модели, которые есть в `MODEL_FOOTPRINTS`. Если в мире появится новый объект,
но его имя не добавлено в этот словарь, для маршрута поиска он не будет
учитываться как препятствие.

При этом `top_down_map_viewer.py` устроен богаче: он может искать `model.sdf`
по `model_search_paths` и извлекать размеры из collision/visual geometry или
mesh. Но эта логика относится к визуализации карты сверху, а не к
`SearchCoveragePlanner`.

## Исключение людей из препятствий

Люди не должны становиться препятствиями для построения зоны покрытия. Если
считать их статическими препятствиями, маршрут поиска начал бы обходить
именно те области, где могут находиться цели.

В `path_planner.py` задан список исключаемых имен:

```python
EXCLUDED_MODEL_NAMES = {
    'grave',
    'man',
    'men',
    'person',
    'people',
    'victim',
    'walker',
}
```

Метод `_is_excluded(model_name, instance_name)` объединяет имя модели и имя
экземпляра, приводит строку к нижнему регистру, заменяет `_` на пробелы и
разбивает текст на токены. Если среди токенов есть слова из
`EXCLUDED_MODEL_NAMES`, модель не добавляется в список препятствий.

Отдельно `grave` исключается и по точному `model_name`. По коду это не
является человеком, но для планировщика эта модель также не рассматривается
как препятствие.

Похожая фильтрация есть в `top_down_map_viewer.py`: viewer исключает модели
людей по параметру:

```text
person_keywords = ['man', 'men', 'person', 'people', 'victim', 'walker']
```

## Преобразование координат

SDF хранит pose моделей в координатах мира Gazebo. Управление дроном и
waypoint-ы работают в локальной системе PX4. Поэтому углы препятствий
переводятся методом:

```python
_map_to_mission_xy(map_x, map_y)
```

По умолчанию используются параметры:

```text
mission_rotation_deg = 90.0
mission_invert_y = True
```

В планировщике применяется обратное преобразование к тому, которое использует
`top_down_map_viewer.py` для отображения mission-local координат на карте.
Сначала точка поворачивается на `-mission_rotation_deg`, затем при
`mission_invert_y = True` инвертируется координата `y`.

Это означает, что корректность маршрута зависит от согласованности параметров
координатного преобразования в миссионном узле и в viewer.

## Построение области покрытия

Автоматический планировщик не берет границы поиска из `mission.py`. Вместо
этого он вычисляет прямоугольные bounds по загруженным препятствиям:

```python
_coverage_bounds()
```

Алгоритм:

1. собираются все `x` и `y` углов всех inflated-препятствий;
2. находится `min_x`, `max_x`, `min_y`, `max_y`;
3. к границам добавляется padding:

```python
padding = max(map_padding, obstacle_margin)
```

Параметр `search_map_padding` тем самым расширяет область покрытия вокруг
известных объектов. В текущих настройках по умолчанию:

```text
search_map_padding = 4.0
search_obstacle_margin = 1.5
```

поэтому bounds расширяются минимум на 4 м.

Ограничение: если значимые области поиска не представлены через известные
препятствия, автоматический планировщик их сам не добавит. Его область
покрытия выводится из SDF-препятствий, а не из отдельного полигона зоны
поиска.

## Построение строк покрытия

После вычисления bounds строятся горизонтальные строки покрытия:

```python
row_spacing = max(coverage_row_spacing, grid_resolution)
```

В `node.py` параметр `coverage_row_spacing` передается из:

```text
search_row_spacing = 7.0
```

Строки создаются от `max_y` к `min_y`:

```python
row_y = max_y
while row_y >= min_y:
    rows.append(row_y)
    row_y -= row_spacing
```

Если последняя строка не дошла до `min_y`, нижняя граница добавляется
отдельно. Таким образом, верхняя и нижняя границы области покрытия тоже
попадают в маршрут.

Для выбора более удобного маршрута создаются несколько вариантов порядка
строк:

- сверху вниз;
- снизу вверх;
- если задан start, от ближайшей к start строки вниз, затем оставшиеся выше;
- если задан start, от ближайшей к start строки вверх, затем оставшиеся ниже.

Для каждого порядка строк дополнительно пробуются две начальные ориентации:

```text
left_to_right = True
left_to_right = False
```

Затем выбирается лучший кандидат. Критерий выбора:

1. минимальная длина первого перехода от стартовой точки к маршруту;
2. при равенстве - меньшая суммарная длина маршрута.

## Свободные интервалы на строке

Каждая строка покрытия разбивается на свободные интервалы методом:

```python
_free_intervals_on_row(y, min_x, max_x, resolution)
```

Алгоритм:

1. строка дискретизируется по `x` от `min_x` до `max_x` с шагом
   `grid_resolution`;
2. для каждой точки `(x, y)` проверяется занятость:

```python
is_free = not self._is_occupied(x, y)
```

3. `_is_occupied()` проверяет, попадает ли точка внутрь любого многоугольника
   препятствия;
4. соседние свободные точки объединяются в непрерывный интервал;
5. слишком короткие интервалы отбрасываются.

Минимальная длина интервала:

```python
min_segment_length = max(waypoint_spacing, resolution)
```

По умолчанию это:

```text
max(search_waypoint_spacing, search_grid_resolution) = max(3.0, 1.0) = 3.0 м
```

Из каждого свободного интервала делается сегмент прохода:

```python
[(start_x, y), (end_x, y)]
```

или в обратном направлении:

```python
[(end_x, y), (start_x, y)]
```

Направление меняется по строкам, за счет чего получается lawnmower/snake
траектория.

## Соединение сегментов через A*

Сами горизонтальные проходы строятся по свободным интервалам, но между
концом предыдущего сегмента и началом следующего может находиться препятствие.
Поэтому добавление сегмента выполняется методом:

```python
_append_segment_with_connector(route, segment)
```

Если маршрут еще пустой, сегмент просто добавляется. Если в маршруте уже есть
точки, планировщик пытается соединить последнюю точку маршрута и начало
нового сегмента через A*:

```python
connector = self.plan(route[-1], segment[0])
```

Метод `plan()` унаследован от `EvacuationRoutePlanner`. Он:

1. строит сетку с шагом `grid_resolution`;
2. размечает клетки как занятые, если их центр попадает в inflated obstacle;
3. переводит start и goal в клетки сетки;
4. если start или goal попали в препятствие, ищет ближайшую свободную клетку;
5. запускает 8-связный A*;
6. использует евклидову эвристику;
7. ограничивает поиск параметром `max_iterations`;
8. восстанавливает путь;
9. сглаживает путь проверками прямой видимости;
10. прореживает waypoint-ы.

Если A* нашел connector минимум из двух точек, к маршруту добавляется весь
connector, кроме первой точки, потому что первая точка совпадает с последней
точкой уже построенного маршрута:

```python
route.extend(connector[1:])
```

Если A* не нашел маршрут, планировщик пробует прямое соединение:

```python
elif self._line_segment_is_free(route[-1], segment[0]):
    route.append(segment[0])
```

Если и A*, и прямая проверка не прошли, сегмент пропускается:

```python
else:
    return
```

Это честное ограничение текущей реализации: недоступный сегмент не вызывает
глобальную перепланировку всего покрытия, а просто не добавляется в итоговый
маршрут.

## A* внутри планировщика

Сеточный A* находится в методе:

```python
_astar(start, goal, occupied)
```

Соседние клетки:

```text
4 ортогональных соседа со стоимостью 1.0
4 диагональных соседа со стоимостью sqrt(2)
```

Эвристика:

```python
math.hypot(first[0] - second[0], first[1] - second[1])
```

Это евклидово расстояние в координатах клеток. A* завершает работу, когда
достигнута goal-клетка или когда число итераций превысило:

```text
max_iterations
```

Для поискового маршрута этот параметр задается как:

```text
search_max_iterations = 20000
```

После A* выполняется сглаживание:

```python
_smooth_path()
```

Сглаживание пытается заменить цепочку промежуточных точек более длинными
прямыми отрезками. Для этого используется проверка:

```python
_line_is_free()
```

Она дискретно проверяет, что отрезок не выходит за границы сетки и не
пересекает занятые клетки.

## Прореживание waypoint-ов

Маршрут после A* и весь итоговый coverage-route прореживаются методом:

```python
_thin_waypoints(points)
```

Смысл прореживания - не отдавать PX4 слишком плотный набор точек. Метод
сохраняет первую и последнюю точки, а промежуточные добавляет только если
расстояние от предыдущей сохраненной точки не меньше:

```python
spacing = max(waypoint_spacing, grid_resolution)
```

Для поискового маршрута по умолчанию:

```text
spacing = max(search_waypoint_spacing, search_grid_resolution)
spacing = max(3.0, 1.0) = 3.0 м
```

В `SearchCoveragePlanner.plan_coverage()` прореженный маршрут возвращается
без стартовой точки `(0.0, 0.0)`, если она была передана как `start` и осталась
первой точкой маршрута:

```python
if start is not None and thinned_route and thinned_route[0] == start:
    return thinned_route[1:]
```

Это сделано потому, что точка старта соответствует takeoff waypoint, а
поисковый маршрут должен начинаться уже с первого рабочего waypoint-а.

## Преобразование 2D route в Waypoint

`SearchCoveragePlanner` возвращает только список XY-точек:

```python
list[tuple[float, float]]
```

В `node.py` они превращаются в `Waypoint` методом:

```python
_waypoints_from_search_route(route_xy, altitude, route_start_xy)
```

Для каждой точки:

- `x` и `y` берутся из route;
- `z` равен `-abs(takeoff_height)`;
- `yaw` оценивается по направлению движения к текущей точке;
- `hold_time` обычно равен `0.0`;
- если точка последняя, ей ставится `search_hold_time_sec`;
- если в точке меняется направление движения больше чем на `0.2` рад,
  waypoint тоже получает `search_hold_time_sec`.

Так дрон не только проходит строки покрытия, но и делает короткие паузы в
точках поворота и в конце маршрута.

## Публикация `mission/search_route`

Миссионный узел создает publisher:

```python
self.search_route_pub = self.create_publisher(
    Float32MultiArray,
    'mission/search_route',
    10,
)
```

Маршрут публикуется сразу после запуска узла:

```python
self._publish_search_route()
```

и затем периодически в основном цикле:

```python
self._maybe_publish_search_route()
```

Период задается параметром:

```text
search_route_publish_period_sec = 2.0
```

Формат сообщения:

```text
[x0, y0, x1, y1, x2, y2, ...]
```

То есть это плоский массив `Float32MultiArray`, где каждая пара чисел
образует одну XY-точку маршрута в mission-local координатах.

Если автоматический план не построился, `search_route_xy` заполняется
точками статического fallback-маршрута:

```python
self.search_route_xy = [
    (waypoint.x, waypoint.y)
    for waypoint in fallback_plan.search_waypoints
]
```

Поэтому viewer получает маршрут и в автоматическом, и в fallback-режиме.

## Отображение маршрута на карте сверху

`top_down_map_viewer.py` подписывается на:

```text
mission/search_route
```

Callback:

```python
_search_route_callback()
```

декодирует `Float32MultiArray` в пары координат, переводит точки из
mission-local XY в координаты карты методом `_mission_to_map_xy()` и затем
рисует полилинию в `_draw_search_route()`.

Маршрут поиска рисуется зеленой линией. Отдельно отмечаются первая, последняя
и часть промежуточных точек, чтобы не перегружать карту.

Параметр визуализации:

```text
draw_search_route = True
```

## Параметры `search_*`

Параметры объявлены и загружаются в `node.py`.

```text
enable_auto_search_route = True
```

Включает автоматическое построение маршрута через `SearchCoveragePlanner`.
Если `False`, всегда используется статический маршрут из `mission.py`.

```text
search_world_sdf_path =
/home/dron/PX4-Autopilot/Tools/simulation/gz/worlds/forest_big.sdf
```

Путь к SDF-файлу мира, из которого берутся `<include>` моделей-препятствий.

```text
search_grid_resolution = 1.0
```

Шаг сетки в метрах для проверки занятости, поиска A* и дискретизации строк
покрытия. Внутри кода значение ограничивается снизу `0.2` м.

```text
search_obstacle_margin = 1.5
```

Запас безопасности вокруг footprint каждого препятствия. Этот запас
добавляется к половине размера объекта по X и Y.

```text
search_map_padding = 4.0
```

Расширение границ области покрытия вокруг всех известных препятствий.

```text
search_row_spacing = 7.0
```

Расстояние между соседними строками lawnmower-маршрута. Фактически
используется `max(search_row_spacing, search_grid_resolution)`.

```text
search_waypoint_spacing = 3.0
```

Минимальный шаг между сохраненными waypoint-ами после прореживания. Также
используется как минимальная длина свободного интервала на строке.

```text
search_hold_time_sec = 1.5
```

Пауза в waypoint-ах, где маршрут меняет направление, и в последней точке
маршрута.

```text
search_route_publish_period_sec = 2.0
```

Период повторной публикации `mission/search_route`, чтобы viewer, запущенный
позже миссионного узла, тоже получил маршрут.

```text
search_mission_rotation_deg = 90.0
search_mission_invert_y = True
```

Параметры преобразования координат между картой Gazebo и mission-local XY.

```text
search_max_iterations = 20000
```

Максимальное число итераций A* для соединительных участков.

## Ограничения текущей реализации

- Автоматический маршрут строится только при наличии препятствий, которые
  удалось загрузить из SDF и сопоставить с `MODEL_FOOTPRINTS`.
- Размеры препятствий для планировщика заданы вручную в `MODEL_FOOTPRINTS`;
  геометрия моделей из `model.sdf` и mesh-файлов в `SearchCoveragePlanner` не
  читается.
- Модели, которых нет в `MODEL_FOOTPRINTS`, игнорируются как препятствия.
- Область покрытия выводится из bounds известных препятствий плюс padding, а
  не из отдельного формального полигона зоны поиска.
- Свободное пространство проверяется дискретно с шагом `search_grid_resolution`;
  мелкие препятствия или узкие проходы меньше разрешения сетки могут быть
  представлены неточно.
- A* используется только для соединения сегментов покрытия. Если конкретный
  сегмент невозможно соединить с уже построенным маршрутом, он пропускается,
  а не запускает полную оптимизацию покрытия.
- Планирование выполняется в 2D по XY. Высота маршрута постоянная и задается
  через `takeoff_height`; вертикальные препятствия, рельеф и реальные
  ограничения динамики БПЛА отдельно не моделируются.
- Люди исключаются из препятствий по ключевым словам. Это правильно для
  поиска целей, но означает, что физический объем модели человека не влияет
  на маршрут.
- Публикуемый `mission/search_route` содержит только XY-точки для
  визуализации; высота, yaw и hold-time существуют в `MissionPlan`, но в этот
  topic не передаются.
- README описывает проект как полет по snake-style маршруту. Это верно для
  fallback и по форме верно для автоматического lawnmower-маршрута, но
  автоматический вариант дополнительно обходит препятствия на соединительных
  участках через A*.

## Что можно написать в дипломе

В работе реализован комбинированный подход к планированию маршрута поиска.
В качестве надежного резервного варианта используется заранее заданная
траектория типа "змейка", покрывающая область лесной сцены с фиксированным
шагом между проходами. Для более адаптивного поведения разработан
автоматический планировщик покрытия `SearchCoveragePlanner`, который строит
маршрут на основе SDF-описания мира Gazebo.

Планировщик извлекает из SDF включенные модели, отбрасывает модели людей и
формирует двумерные footprint-ы известных статических препятствий. Размеры
таких препятствий задаются словарем `MODEL_FOOTPRINTS`, после чего каждый
footprint расширяется на заданный запас безопасности. По полученным
препятствиям вычисляется область покрытия, разбивается на горизонтальные
строки, а на каждой строке находятся свободные интервалы, не пересекающие
препятствия. Из этих интервалов формируются проходы lawnmower-типа.

Для соединения отдельных проходов используется сеточный алгоритм A* с
8-связной окрестностью и евклидовой эвристикой. Найденные соединительные
маршруты сглаживаются проверками прямой видимости и прореживаются, чтобы
получить набор waypoint-ов, пригодный для следования в режиме PX4 Offboard.
Если автоматический маршрут построить невозможно, система автоматически
возвращается к статическому маршруту из `mission.py`, что повышает
устойчивость демонстрационного сценария.

Итоговый маршрут публикуется в ROS 2 topic `mission/search_route` в виде
последовательности XY-точек. Этот topic используется картой сверху для
визуального контроля того, какой маршрут поиска был сформирован и по какой
области должен пройти БПЛА.
